# Telegram Quiz Bot

Асинхронный Telegram-бот для опросов/тестов с хранением вопросов, вариантов ответов и результатов пользователей в PostgreSQL.

## Возможности

- вопросы и варианты ответов хранятся в базе данных;
- вопрос может содержать `photo_url` или `photo_file_id` Telegram;
- каждый ответ пользователя сохраняется в таблицу `user_answers`;
- после выбора ответа сообщение с текущим вопросом удаляется из чата, чтобы не оставались уже отвеченные вопросы и варианты;
- пользователи, ответившие на все вопросы правильно, сохраняются в `successful_users`;
- после завершения бот отправляет сообщение `Опрос завершён. Спасибо за ваши ответы!`;
- повторное прохождение запрещено после любого полностью завершённого опроса;
- в меню команд Telegram отображается пункт `Перезапуск`, который вызывает команду `/start`;
- опрос можно закрыть allowlist-списком Telegram-пользователей для закрытого тестирования;
- асинхронная обработка Telegram-событий и пул соединений PostgreSQL для высокой нагрузки;
- контейнер запускается без root, с `read_only`, `cap_drop` и `no-new-privileges`.

## Быстрый запуск

1. Создайте бота у [@BotFather](https://t.me/BotFather) и получите токен.
2. Скопируйте переменные окружения:

   ```bash
   cp .env.example .env
   ```

3. Укажите `BOT_TOKEN` и, при необходимости, измените пароль PostgreSQL:

   ```env
   BOT_TOKEN=123456:your_real_token
   POSTGRES_PASSWORD=long-random-password
   DATABASE_URL=postgresql://quiz:long-random-password@postgres:5432/quiz_bot
   ```

4. Запустите среду:

   ```bash
   docker compose up -d --build
   ```

5. Загрузите пример вопросов:

   ```bash
   docker compose run --rm bot python -m app.seed app/data/questions.example.json
   ```

   Если выполняете загрузку с хоста, а не внутри Docker Compose, используйте адрес PostgreSQL, опубликованный на `127.0.0.1`:

   ```bash
   python -m app.seed app/data/questions.example.json --database-url postgresql://quiz:long-random-password@127.0.0.1:5432/quiz_bot
   ```

   Ошибка `Temporary failure in name resolution` обычно означает, что команда запущена вне сети Docker Compose с `DATABASE_URL`, где хост указан как `postgres`.

6. Откройте бота в Telegram и отправьте `/start`.

## Shell-меню для консольных команд

Для запуска сервисных команд без ручного набора длинных команд используйте интерактивное shell-меню:

```bash
./scripts/bot-menu.sh
```

В меню можно:

- загрузить вопросы из JSON; перед запуском загрузки меню обязательно спросит путь к файлу с вопросами;
- посмотреть отчёты по попыткам, ответам и победителям в формате `table` или `csv`;
- очистить тестовые данные с дополнительным подтверждением;
- управлять allowlist-списком пользователей закрытого тестирования;
- переключить режим запуска между Docker Compose и локальным Python.

По умолчанию меню запускает команды через Docker Compose. При загрузке вопросов файл выбирается интерактивно и монтируется в контейнер только для чтения, поэтому можно указать JSON-файл из любого каталога на хосте. Для локального режима меню предложит указать `DATABASE_URL` или взять его из окружения/`.env`.

> Если пользователь нажал `/start` до загрузки вопросов, бот создаёт пустую незавершённую попытку.
> После загрузки JSON такая попытка автоматически пересчитывается при следующем `/start`,
> поэтому сообщение «Опрос пока не содержит активных вопросов» не должно оставаться для уже загруженного опроса.

## Редактирование вопросов

Самый удобный способ — отредактировать JSON-файл с вопросами и загрузить его командой `app.seed`.
Первый активный вопрос по `sort_order` считается приветствием: бот отправляет его без префикса `Вопрос 1`.
Нумерация начинается со следующего вопроса, поэтому второй вопрос в базе отображается как `Вопрос 1`.
Формат файла:

```json
[
  {
    "sort_order": 1,
    "text": "Текст вопроса",
    "photo_url": "https://example.com/image.jpg",
    "photo_file_id": null,
    "options": [
      {"text": "Правильный ответ", "is_correct": true, "sort_order": 1},
      {"text": "Неправильный ответ", "is_correct": false, "sort_order": 2}
    ]
  }
]
```

Если фото уже загружено в Telegram, можно указать `photo_file_id`; иначе используйте публичный HTTPS URL в `photo_url`.
`photo_url` должен вести прямо на изображение; если Telegram не сможет отправить фото, бот запишет предупреждение в лог и отправит вопрос обычным текстом с вариантами ответа.


## Закрытое тестирование

По умолчанию список доступа пуст, поэтому опрос открыт для всех пользователей.
Как только в список добавлен хотя бы один Telegram user ID, пройти опрос и отвечать на вопросы смогут только пользователи из этого списка.

Добавить или обновить пользователя в списке доступа:

```bash
docker compose run --rm bot python -m app.maintenance allow-users add 123456789 --username tester --note "закрытый тест"
```

Посмотреть текущий список:

```bash
docker compose run --rm bot python -m app.maintenance allow-users list
```

Удалить пользователя из списка:

```bash
docker compose run --rm bot python -m app.maintenance allow-users remove 123456789
```

Полностью очистить список и снова открыть опрос для всех:

```bash
docker compose run --rm bot python -m app.maintenance allow-users clear --yes
```

Если запускаете команды с хоста, передайте `--database-url` перед названием команды, например:

```bash
python -m app.maintenance --database-url postgresql://quiz:long-random-password@127.0.0.1:5432/quiz_bot allow-users list
```

## Просмотр результатов

В проекте есть консольная команда `app.report`, которая читает PostgreSQL и показывает результаты без доступа к Telegram.

Посмотреть все попытки пользователей: кто начал опрос, сколько вопросов уже отвечено и завершена ли попытка:

```bash
docker compose run --rm bot python -m app.report attempts
```

Посмотреть все сохранённые ответы пользователей с текстом вопроса, выбранным вариантом и признаком правильности:

```bash
docker compose run --rm bot python -m app.report answers
```

Посмотреть список победителей — пользователей, которые ответили правильно на все вопросы:

```bash
docker compose run --rm bot python -m app.report winners
```

Для выгрузки в CSV добавьте `--format csv`, например:

```bash
docker compose run --rm bot python -m app.report winners --format csv > winners.csv
```

Если запускаете отчёт не через Docker Compose, укажите `DATABASE_URL` в окружении или передайте подключение явно через `--database-url`.
Для запуска с хоста вместо Docker-сервисного имени `postgres` используйте `127.0.0.1`, например `postgresql://quiz:...@127.0.0.1:5432/quiz_bot`.

## Очистка базы перед тестированием

Для повторного тестового прохождения можно очистить пользовательские данные без удаления вопросов:

```bash
docker compose run --rm bot python -m app.maintenance reset-db --yes
```

Команда удаляет попытки, сохранённые ответы и победителей, но оставляет вопросы и варианты ответов.
Если нужно полностью пересобрать тестовую базу вместе с вопросами, добавьте `--with-questions`:

```bash
docker compose run --rm bot python -m app.maintenance reset-db --with-questions --yes
```

Если запускаете команду с хоста, укажите `DATABASE_URL` в окружении или передайте `--database-url`.
Флаг `--yes` обязателен, чтобы случайно не удалить тестовые данные.

## Структура базы данных

- `questions` — активные вопросы опроса;
- `answer_options` — варианты ответов и признак правильности;
- `quiz_attempts` — попытки прохождения пользователей;
- `user_answers` — все ответы пользователей;
- `successful_users` — пользователи, ответившие правильно на все вопросы;
- `quiz_allowed_users` — allowlist пользователей для закрытого тестирования.

## Нагрузка и безопасность

- Бот написан на `aiogram` и использует неблокирующий I/O.
- PostgreSQL используется через `asyncpg` с настраиваемым пулом `DB_POOL_MIN_SIZE`/`DB_POOL_MAX_SIZE`.
- Уникальные ограничения БД защищают от повторного ответа на один вопрос в рамках попытки.
- Все сообщения бота отправляются с защитой Telegram `protect_content=True`, чтобы ограничить пересылку и сохранение контента пользователями.
- Callback-ответы проверяются по активной попытке пользователя и ID вопроса/варианта.
- Токены и пароли не хранятся в репозитории — используйте `.env`.
- Для публичного продакшена рекомендуется запускать контейнеры за firewall, регулярно обновлять образы и использовать отдельного DB-пользователя с минимальными правами.
